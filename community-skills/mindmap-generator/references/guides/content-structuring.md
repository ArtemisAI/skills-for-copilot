# 内容结构化策略

本文档提供将分析后的内容组织成思维导图的详细策略。

## 核心原则

### 自顶向下

从整体到细节构建思维导图：
1. 确定一级标题（核心主题）
2. 识别主要维度（二级标题）
3. 逐层展开细节（三到六级）

### 结构化分类

使用一致的分类标准：
- **功能维度**: 做什么、怎么做、为什么做
- **时间维度**: 历史、现状、未来
- **空间维度**: 整体、部分、细节
- **逻辑维度**: 定义、特征、应用、问题

## 层级设计

### 一级标题（根节点）

- 全文唯一
- 简洁明确（5-10 字）
- 准确概括主题

示例：
- `# React Hooks`
- `# 微服务架构`
- `# Git 工作流程`

### 二级标题（主要分支）

- 建议 3-7 个
- 覆盖核心维度
- 并列关系

常见模式：
- **概念类**: 定义 → 特征 → 原理 → 应用
- **技术类**: 概述 → 组件 → 实现 → 最佳实践
- **流程类**: 准备 → 执行 → 验证 → 优化
- **知识类**: 基础 → 核心 → 进阶 → 实战

### 三到六级标题（详细展开）

- 每级不超过 7 个同级节点
- 避免过深层级（建议 ≤5 级）
- 保持逻辑递进

## 内容组织技巧

### 并列关系

同级节点应该：
- 分类标准一致
- 内容互斥
- 合计完整

示例（正确）:
```markdown
## 前端技术栈
### React
### Vue
### Angular
```

示例（错误 - 分类标准不一致）:
```markdown
## 技术栈
### React
### 数据库
### 服务器
```

### 递进关系

父子节点应该：
- 逻辑递进（从抽象到具体）
- 包含关系（父节点包含子节点）
- 层次分明

示例:
```markdown
## 状态管理
### useState
#### 基本用法
#### 注意事项
### useReducer
#### 使用场景
#### 最佳实践
```

## 内容优化

### 节点简洁化

每个节点应该：
- 1-2 行文字
- 关键词优先
- 避免冗长描述

优化前:
```markdown
### React 是由 Facebook 开发的一个用于构建用户界面的 JavaScript 库
```

优化后:
```markdown
### React 核心特性
```

### 结构扁平化

避免过深嵌套：

优化前（6 级深度）:
```markdown
# 主题
## 分支
### 子分支
#### 详细内容
##### 更详细
###### 太深了
```

优化后（4 级深度）:
```markdown
# 主题
## 分支
### 核心要点1
#### 详细说明
### 核心要点2
#### 详细说明
```

### 平衡性调整

检查并调整：
- 各分支节点数量是否均衡
- 是否有分支过于单薄
- 是否有分支过于臃肿

## 质量检查清单

结构化完成后，检查：

- [ ] 一级标题唯一且准确
- [ ] 二级标题 3-7 个，覆盖核心维度
- [ ] 层级深度 ≤5 级
- [ ] 同级节点分类标准一致
- [ ] 父子节点逻辑递进
- [ ] 每个节点内容简洁（1-2 行）
- [ ] 避免单子节点（考虑合并）
- [ ] 避免同级节点过多（考虑分组）
